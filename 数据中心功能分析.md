### 1.2 无法显示/有问题的数据 ❌

目前**所有定义的字段都可以正常显示**，但存在以下潜在问题：

#### **数据准确性问题**
1. **物理合理性检查**
   ```kotlin
   private fun isPlausible(s: DriveStats): Boolean {
       val durH = s.durationMinutes / 60f
       return !(durH > 0 && s.totalDistanceKm / durH > 150f)
   }
   ```
   - 过滤掉平均速度 > 150 km/h 的记录（可能是数据损坏）

2. **数据缺失场景**
   - C3 没有segment数据 → 显示空状态
   - qlog文件损坏 → 该天数据为0
   - 脚本未部署 → 无法同步

#### **数据完整性依赖**
- **必须条件**：
  - C3上存在 `/data/media/0/realdata/*--*/rlog.zst` 文件
  - `calc_drive_stats.py` 脚本正确部署
  - SSH连接正常

## 二、数据统计方式

### 2.1 数据来源链路

```
C3端 realdata (qlog)
         ↓
calc_drive_stats.py 解析
         ↓
C3缓存: /data/appdata/drive_stats.json
         ↓
SSH传输
         ↓
Android App: Room 数据库 (SQLite)
         ↓
聚合计算 (aggregate)
         ↓
UI显示
```

### 2.2 统计脚本 `calc_drive_stats.py`

#### **位置**
- 本地：`app/src/main/assets/calc_drive_stats.py`
- C3端：`/data/openpilot/c3_scripts/calc_drive_stats.py`

#### **功能**
1. 扫描 `/data/media/0/realdata/` 下所有segment目录
2. 解析每个segment的 `rlog.zst` 文件
3. 提取关键字段：
   - 里程：`carState.vEgo` 积分计算
   - 辅助状态：`controlsState.enabled`
   - 事件：`carEvents` (takeover, collision, tailgating等)
4. 按天聚合统计
5. 输出JSON数组到stdout

#### **运行模式**
```bash
# 全量扫描（首次使用）
python3 calc_drive_stats.py --full

# 增量扫描（仅扫描新segment）
python3 calc_drive_stats.py --incremental
```

### 2.3 增量标记机制

#### **增量追踪文件**
```
/data/appdata/last_sync.txt
```
- 记录上次扫描的最后一个segment ID
- 增量模式只处理比此ID更新的segment

#### **缓存文件**
```
/data/appdata/drive_stats.json
```
- 存储C3端累积的全部统计数据
- 格式：JSON数组，每个元素代表一天的数据
- 作用：App重装后快速恢复历史数据（秒级）

## 五、数据聚合计算

### 5.1 聚合字段计算公式

| 字段 | 计算公式 | 说明 |
|------|----------|------|
| `assistedPercent` | `(assisted / total) * 100` | 智驾里程占比 |
| `manualPercent` | `(manual / total) * 100` | 人工里程占比 |
| `durationRatioPercent` | `(assistedDur / totalDur) * 100` | 智驾时长占比 |
| `avgSpeedKmh` | `total / (duration / 60)` | 平均速度 |
| `takeoversPerKkm` | `takeovers / (total / 1000)` | 千公里接管率 |
| `safetyScore` | `100 - 接管惩罚 - 预警惩罚` | 安全评分 |

### 5.2 安全评分算法

```kotlin
private fun calculateScore(list: List<DriveStats>): Int {
    if (list.isEmpty()) return 0
    val total = list.sumOf { it.totalDistanceKm.toDouble() }.toFloat()
    val takeovers = list.sumOf { it.takeovers }
    val warnings = list.sumOf {
        it.collisionWarning + it.tailgating + it.leadCarEmergencyBrake +
        it.leadCarSlow + it.startReminder + it.laneChangeAssist
    }
    var score = 100
    if (total > 0) {
        score -= (takeovers / (total / 1000f) * 2).toInt()  // 接管惩罚
        score -= (warnings / (total / 100f)).toInt()        // 预警惩罚
    }
    return score.coerceIn(0, 100)
}
```

**评分规则**：
- 基础分：100分
- 接管惩罚：每千公里接管1次 → 扣2分
- 预警惩罚：每百公里预警1次 → 扣1分
- 最终分数：0~100分

object SyncStateHolder {

**特点**：
- 全局单例，进程级状态
- 离开数据中心页面也继续运行
- 使用 `StateFlow` 响应式更新UI

### 6.2 同步状态流转

```
IDLE (空闲)
  ↓
CONNECTING (连接中)
  ↓
CHECKING (检查数据)
  ↓
COMPUTING (统计中)
  ↓
PARSING (解析中)
  ↓
SUCCESS (成功) / ERROR (失败)
  ↓
IDLE
```

### 6.3 同步阶段提示

| 阶段 | 提示文本 | 耗时 |
|------|----------|------|
| CONNECTING | "正在检查脚本…" | <1s |
| CONNECTING | "正在部署统计脚本到 C3…" | 1~3s |
| CHECKING | "正在从 C3 缓存恢复数据…" | 1~2s |
| PARSING | "正在加载历史数据（秒级）…" | 1~3s |
| COMPUTING | "正在全量统计（约 1~3 分钟）…" | 60~180s |
| COMPUTING | "正在增量统计（约 5~30 秒）…" | 5~30s |
| CHECKING | "正在检查新数据…" | <1s |

## 八、性能优化

### 8.1 数据库优化

```kotlin
@Dao
interface DriveStatsDao {
    @Query("SELECT * FROM drive_stats WHERE date BETWEEN :start AND :end ORDER BY date DESC")
    suspend fun getBetween(start: String, end: String): List<DriveStats>
    
    @Query("SELECT MIN(date) FROM drive_stats")
    suspend fun getEarliestDate(): String?
    
    @Query("SELECT MAX(date) FROM drive_stats")
    suspend fun getLatestDate(): String?
}
```

**索引**：
- `date` 作为主键自动建立索引
- 日期范围查询非常高效

### 8.2 缓存策略

1. **C3端缓存** (`drive_stats.json`)
   - 避免重复解析qlog
   - App重装后秒级恢复

2. **增量标记** (`last_sync.txt`)
   - 只处理新segment
   - 增量同步仅需5~30秒

3. **物理合理性过滤**
   - 在聚合阶段过滤异常数据
   - 避免显示不合理的统计结果

### 9.1 常见错误场景

## 总结

### ✅ 优点
1. **数据完整性**：所有定义的指标都能正常显示
2. **智能同步**：自动判断全量/增量/缓存恢复策略
3. **后台运行**：离开页面不影响同步
4. **数据持久化**：C3端+Android端双重缓存
5. **物理校验**：过滤不合理数据
6. **导入导出**：支持数据迁移和备份

### ⚠️ 注意事项
1. 首次同步耗时较长（1~3分钟）
2. 依赖C3的realdata目录有数据
3. 需要SSH连接稳定
4. 统计脚本必须正确部署

### 📊 数据存储占用
- **C3端**：~100KB（100天统计缓存）
- **Android端**：~73KB（365天数据库）
- **总计**：<200KB（非常轻量）
